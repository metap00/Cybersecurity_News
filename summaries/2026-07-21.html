'LegacyHive' Windows Zero-Day Drops With No Patch — Grants Admin Access on All Supported Versions
🔍 What happened: Security researcher Nightmare Eclipse publicly released a working proof-of-concept exploit called LegacyHive that exploits an unpatched Windows User Profile Service vulnerability to escalate privileges to administrator on all supported Windows versions.
👤 Who's affected: Every organization running Windows 10, Windows 11, or Windows Server 2016/2019/2022 — including systems fully patched as of July 2026 Patch Tuesday — is exposed with no vendor fix available.
✅ What to do: Enforce least-privilege access policies, restrict local logon rights, deploy endpoint detection rules for anomalous User Profile Service registry activity, and monitor for out-of-band advisories from Microsoft.
🎯 ATT&CK: [Privilege Escalation] T1068 · Exploitation for Privilege Escalation
The LegacyHive PoC uses a carefully timed path-switching race condition to force the Windows User Profile Service to mount an administrator's registry hive under a standard user account, granting immediate system-level read/write access without requiring any credentials.
▶ Full analysis & beginner explanation

The LegacyHive exploit targets the Windows User Profile Service (ProfSvc), a core Windows component responsible for loading each user's registry hive (NTUSER.DAT) and configuring their environment at logon. The attack abuses a race condition in how ProfSvc resolves file paths during a logon event: by rapidly switching a path reference at precisely the right moment, an attacker running as a standard low-privilege user can trick the service into mounting a high-privilege user's registry hive — including a local administrator's — under their own account context. Once mounted, the attacker gains full read/write access to the administrator's registry profile and can escalate to SYSTEM.

What makes LegacyHive uniquely alarming is that it works on fully patched systems as of July 14, 2026 Patch Tuesday, despite Microsoft having shipped a record-breaking 570+ CVE fixes that day. The vulnerability was not included in any of those patches. Nightmare Eclipse — a prolific researcher behind previous Windows zero-day disclosures including RoguePlanet, BlueHammer, and UnDefend — released the PoC publicly, although they reportedly stripped the most directly weaponizable components to limit immediate exploitation by low-skill attackers.

Microsoft has not yet assigned a CVE or published a security advisory. Organizations should treat this as an emergency configuration review: audit who holds local logon rights on sensitive systems, deploy EDR rules watching for anomalous ProfSvc registry load events, and prepare incident response runbooks for rapid patch deployment once Microsoft responds. Temporary mitigations include blocking interactive local logon for non-admin accounts on critical servers and monitoring Windows Event ID 1511 and 1514 for unexpected profile service activity.

🎓 For beginners: A privilege escalation exploit is like finding a trick at a hotel to swap your regular room key for the master key. LegacyHive fools Windows into handing a normal user account the master key (the administrator's settings and permissions) by briefly confusing the part of Windows that sets up user environments at login — giving the attacker full control of the computer without needing to know any password.
No CVE assigned · Under active investigation — Microsoft has not yet published a security advisory or patch for LegacyHive; affected scope is every supported Windows version.

WP2Shell: WordPress Core Pre-Auth RCE Chain (CVE-2026-60137 + CVE-2026-63030) Under Mass Exploitation
🔍 What happened: Two chained WordPress core vulnerabilities — a SQL injection and a REST API route confusion flaw — allow any unauthenticated attacker to achieve remote code execution on WordPress sites, with 24+ public PoC exploits already in wide use just days after disclosure.
👤 Who's affected: Tens of millions of WordPress websites running version 6.8 or later (SQL injection) or 6.9+ (full RCE chain), including any site that hasn't applied emergency patches or allowed automatic updates.
✅ What to do: Update WordPress immediately to 7.0.2 (or backport releases 6.9.5 / 6.8.6), enable auto-updates if not already on, and deploy a WAF rule blocking requests to the REST batch endpoint matching known WP2Shell patterns.
🎯 ATT&CK: [Initial Access] T1190 · Exploit Public-Facing Application
Attackers target internet-exposed WordPress sites by sending crafted requests to the public REST API batch endpoint, chaining the SQL injection in CVE-2026-60137 with the route confusion in CVE-2026-63030 to write and then execute arbitrary code without any login credentials.
▶ Full analysis & beginner explanation

The WP2Shell attack chain combines two distinct WordPress core flaws. CVE-2026-60137 is a SQL injection in the WP_Query class caused by improper sanitization of the author__not_in parameter, present since WordPress 6.8. CVE-2026-63030 is a REST API batch-route confusion vulnerability in WordPress 6.9+ where an unauthenticated request can reach code paths normally protected by authentication, particularly when a persistent object cache is not configured. When chained together, an attacker first uses the SQL injection to write a malicious payload into the WordPress database, then triggers execution through the REST batch endpoint — achieving full remote code execution with zero credentials required.

WordPress disclosed both vulnerabilities on July 17, 2026. Within hours, multiple proof-of-concept exploits appeared publicly, and by July 19, VulnCheck had verified more than two dozen unique PoCs actively targeting unpatched sites. Cloudflare, Patchstack, and other WAF providers deployed emergency protection rules. WordPress classified this as its highest-ever severity issue and activated forced automatic updates — but auto-update propagation has delays, and many site operators explicitly disable it, leaving substantial exposure. Mass scanning campaigns targeting the REST batch endpoint have been observed across the internet.

WordPress.org released fixes in version 7.0.2 with backports to 6.9.5 and 6.8.6. Site administrators should verify their installed version, check error logs for REST API batch endpoint abuse, rotate admin credentials on any site that may have been compromised, and audit for webshells or unauthorized admin accounts. Hosting providers should also scan customer sites en masse.

🎓 For beginners: WordPress powers roughly 40% of all websites. WP2Shell is like discovering two unlocked back doors into a store: the first door (SQL injection) lets anyone rearrange the store's stockroom database, and the second door (REST API confusion) lets anyone walk straight into the manager's office to take over. Together, with no key needed at all, an attacker can take complete control of the website — deleting content, stealing data, or installing hidden malware.
CVE-2026-60137 + CVE-2026-63030 · CVSS: ~9.8/10 — Both CVEs are pre-authentication, require no user interaction, and the chained attack achieves arbitrary remote code execution, meeting the criteria for the maximum exploitability rating.

CISA Adds Exploited SharePoint Zero-Day CVE-2026-58644 to KEV — CVSS 9.8 Unauthenticated RCE
🔍 What happened: Microsoft patched CVE-2026-58644, a critical deserialization RCE in SharePoint Server (CVSS 9.8), on July 14 Patch Tuesday — but later confirmed it was already being exploited as a zero-day in the wild before the fix shipped; CISA added it to KEV with a July 19 federal patch deadline.
👤 Who's affected: Organizations running on-premises Microsoft SharePoint Server Subscription Edition, 2019, or 2016 — especially government agencies, defense contractors, and enterprises that have not yet applied July Patch Tuesday updates.
✅ What to do: Apply the July 2026 Patch Tuesday updates immediately; audit IIS machine keys and server logs for signs of compromise; search for unusual scheduled tasks, new local accounts, or registry persistence indicators that may signal post-exploitation activity.
🎯 ATT&CK: [Initial Access] T1190 · Exploit Public-Facing Application
Threat actors are targeting internet-facing SharePoint Server deployments by sending specially crafted requests that trigger the deserialization pipeline with attacker-controlled data, achieving unauthenticated remote code execution and then stealing IIS machine keys for further persistence.
▶ Full analysis & beginner explanation

CVE-2026-58644 is a deserialization of untrusted data vulnerability in Microsoft SharePoint Server. Deserialization flaws occur when an application converts incoming data back into executable objects without first verifying that the data is safe — an attacker can craft malicious serialized objects that, when processed, execute arbitrary commands with the privileges of the application. In SharePoint's case, this flaw is reachable without authentication, meaning no login credentials are required to trigger the exploit. The CVSS score of 9.8 reflects this near-worst-case combination of remote exploitability, no authentication requirement, and code execution outcome.

Microsoft patched the vulnerability on July 14, 2026, as part of its record July Patch Tuesday. However, Microsoft subsequently updated the advisory to confirm that CVE-2026-58644 was exploited in the wild before the patch was available — classifying it as a true zero-day. Observed post-exploitation activity includes theft of IIS machine keys (which can be used to forge authentication tokens and decrypt sensitive data), deployment of persistence mechanisms, and further deserialization-based malware staging. CISA added the vulnerability to its Known Exploited Vulnerabilities catalog and ordered all Federal Civilian Executive Branch agencies to apply the patch by July 19, 2026.

SharePoint Server is widely deployed in government, defense, and enterprise environments, making this an extremely high-value target. Organizations should immediately apply the patch, rotate IIS machine keys, and conduct a thorough hunt for indicators of compromise: unusual child processes spawned by w3wp.exe, new scheduled tasks or services, and outbound connections from the SharePoint server to unusual destinations. Cloud-based SharePoint Online deployments are not affected.

🎓 For beginners: Deserialization is like a restaurant that automatically cooks and serves whatever "recipe card" a customer hands them without reading it first. CVE-2026-58644 lets an attacker hand SharePoint a recipe card containing malware instructions — and SharePoint blindly executes them, giving the attacker full control of the server. No login required: just sending the right malicious request over the internet is enough.
CVE-2026-58644 · CVSS: 9.8/10 — Unauthenticated network-exploitable remote code execution with no user interaction required; the highest exploitability tier in CVSS 3.1 scoring.

Iran Exploited SS7 Telecom Vulnerabilities and Ad-Tech Metadata to Track US Military Smartphones
🔍 What happened: Iranian intelligence actors abused well-known SS7 signaling vulnerabilities in the global cellular roaming infrastructure and commercial advertising technology metadata to locate and monitor US military personnel's smartphones during the Iran War build-up — exploiting decades-old telecoms architecture for real-time location intelligence.
👤 Who's affected: US military personnel and government officials operating commercial mobile phones in or near the Middle East; the technique is applicable to any individual whose phone number is known to an adversary with access to SS7 infrastructure.
✅ What to do: Military and high-value individuals should use DoD-approved secure communication devices, disable SS7-vulnerable network features, avoid commercial cellular in sensitive operational contexts, and use satellite or encrypted-over-WiFi communication alternatives where possible.
🎯 ATT&CK: [Collection] T1430 · Location Tracking
Iranian intelligence exploited SS7 protocol weaknesses — specifically the SendRoutingInfo and ProvideSubscriberInfo messages in the SS7 MAP layer — to query the global cellular signaling network for real-time GPS-resolution location data on US military personnel's phone numbers without the knowledge of their carriers.
▶ Full analysis & beginner explanation

SS7 (Signaling System 7) is a 1975-era protocol suite that phone networks worldwide use to coordinate how calls and texts travel between carriers — it enables your phone to receive calls while roaming abroad. SS7 was designed for a small club of trusted national telecoms; it has no authentication between network nodes, meaning any carrier (or entity with illicit access to carrier switching equipment) can send queries to the global network. Iran reportedly exploited this architectural flaw by sending SS7 MAP-layer messages — specifically SendRoutingInfo and ProvideSubscriberInfo queries — that caused the global cellular network to return real-time location data for targeted US military phone numbers, accurate to cellular-tower resolution and sometimes GPS-level precision.

Additionally, Iranian actors leveraged commercial advertising technology: mobile apps constantly broadcast device identifiers (advertising IDs, IMEIs) and location signals to ad networks. By purchasing access to location data brokers or exploiting ad-network APIs, adversaries can track the movements of specific devices without touching any telecom infrastructure. The convergence of these two techniques — the legacy SS7 flaw and the commercial ad-tech ecosystem — provided Iranian intelligence with persistent, near-real-time location tracking of US military personnel. This methodology has been documented by telecoms security researchers for over a decade, but the Iran War context represents one of the most significant known applications of SS7 exploitation against military targets.

🎓 For beginners: When your phone travels abroad, it announces itself to foreign phone networks so they can route your calls home. SS7 is the global "phone directory" system that makes this work — but it was built in 1975 when only a handful of phone companies existed and everyone trusted each other. Iran exploited the fact that SS7 has no security checks: just like anyone can look up an address in an old phone book, they could query the global system and get the real-time location of any US military phone number they knew.

💡 Today I Learned: SS7 Signaling Attack ▶
SS7 (Signaling System 7) is a 1975-era protocol suite that underpins global phone networks, enabling calls and texts to be routed between carriers worldwide — it was designed when only a small number of national telephone companies existed, and mutual trust was simply assumed with no authentication built in. Because any entity with access to the SS7 network (a rogue employee, a compromised carrier, or a nation-state intelligence service) can send unauthenticated queries to the global signaling infrastructure, they can retrieve real-time location data, intercept SMS messages (including two-factor authentication codes), and eavesdrop on calls for any phone number in the world. Iran's exploitation of SS7 to track US military phones is a real-world demonstration of why security researchers have warned about this flaw for over a decade — and why governments and sensitive organizations increasingly mandate satellite communications and end-to-end encrypted messaging apps that route around the vulnerable SS7 layer entirely.
📖 Glossary: SS7 · Vishing · SSO · PHI · REST API · FCEB · Supply Chain Attack ▶
SS7 — Signaling System 7; a 1975-era protocol suite that global phone networks use to route calls and texts between carriers, notorious for lacking authentication and enabling location tracking and call interception by anyone with network access.
Vishing — Voice phishing; a social engineering attack where criminals call targets by phone, impersonating IT support, executives, or financial institutions to trick them into revealing credentials or taking actions that compromise security.
SSO — Single Sign-On; an authentication technology that lets a user log in once to access many connected applications, widely used in enterprises via providers like Microsoft Entra (formerly Azure AD), Okta, or Google Workspace.
PHI — Protected Health Information; any individually identifiable health data regulated under US HIPAA law, including diagnoses, treatment records, health insurance details, and SSNs when linked to health context.
REST API — Representational State Transfer Application Programming Interface; a standard approach for web applications to exchange data over HTTP, widely used by modern software and a frequent attack surface when endpoints lack proper authentication or input validation.
FCEB — Federal Civilian Executive Branch; the US civilian government agencies (such as DHS, State, Education) subject to CISA's binding operational directives and Known Exploited Vulnerabilities patching mandates.
Supply Chain Attack — A cyberattack that targets a vendor, software provider, or third-party service partner as a pathway to reach and compromise the intended victim organization, exploiting trusted relationships rather than attacking the target directly.
