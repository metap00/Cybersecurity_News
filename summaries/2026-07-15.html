ESET: 11 Forgotten Microsoft-Signed UEFI Shims Let Attackers Bypass Secure Boot on Any UEFI Machine
🔍 What happened: ESET researchers found 11 old but still-valid UEFI shim bootloaders (version 0.9 and below), signed by Microsoft's UEFI CA 2011 certificate, that can be abused on any machine trusting that CA to bypass Secure Boot and execute arbitrary pre-OS code.
👤 Who's affected: Any UEFI-based system trusting the Microsoft Corporation UEFI CA 2011 third-party certificate — including most enterprise and consumer PCs — is vulnerable to bootkit deployment if an attacker has physical or high-privilege OS-level access.
✅ What to do: Microsoft revoked the vulnerable shims in the June 9, 2026 Patch Tuesday; ensure that update is applied, as it adds the shims to the UEFI Secure Boot Forbidden Signature Database (DBX).
🎯 ATT&CK: [Defense Evasion] T1542.003 · Pre-OS Boot: Bootkit
Attackers load a vulnerable signed UEFI shim during system boot — before the OS initialises — to execute untrusted code that survives OS reinstalls and can deploy persistent bootkits like BlackLotus or HybridPetya.
▶ Full analysis & beginner explanation

ESET's research exposed a systemic gap in UEFI Secure Boot's revocation model: old shim bootloaders that were signed legitimately years ago but have since been replaced by secure versions were never explicitly blacklisted. Because the signing certificate is still trusted, an attacker can "bring their own vulnerable shim" — a BYOVD-style technique applied to the firmware layer — and use it to bypass Secure Boot entirely.

The 11 affected shims are all at version 0.9 or below, used in older Linux distributions. They are tracked under CVE-2026-8863 and CVE-2026-10797. While ESET disclosed the issues responsibly in February 2026 and Microsoft revoked them in the June Patch Tuesday, many organisations have not yet applied that update, leaving them exposed. The vulnerable shims can enable deployment of known UEFI bootkits including Bootkitty, HybridPetya, and BlackLotus.

This research highlights a critical gap in how UEFI firmware trust is managed: the signing infrastructure is sound, but its revocation list (DBX) is only as effective as the patch cadence of the organisations relying on it. ESET has called for a more systematic approach to shim lifecycle management across the ecosystem.

🎓 For beginners: Secure Boot is like a bouncer at your computer's door that checks everyone's ID before the computer fully wakes up. These old "IDs" (shim files) were legitimate once, but their owners did dangerous things. The bouncer should have added them to a banned list, but didn't — so attackers can still use those old IDs to sneak past security before your antivirus even loads.
CVE-2026-8863 & CVE-2026-10797 — UEFI Secure Boot bypass via unsigned-code execution through trusted-but-vulnerable signed shims; patched in Microsoft's June 2026 Patch Tuesday DBX update.

Repositories clone the names, branding, and README styles of legitimate security and developer tools so that victims searching for trusted software find and download the malicious version instead.
▶ Full analysis & beginner explanation

Arctic Wolf researchers identified a large-scale brand impersonation campaign on GitHub targeting security practitioners and developers — the very people most likely to be guarded against social engineering. 💡 Today I Learned: Server-Side Request Forgery (SSRF) ▶
Server-Side Request Forgery (SSRF) is an attack where an adversary tricks a web server or appliance into making HTTP requests on their behalf — to internal systems the attacker cannot reach directly from the internet. In today's SonicWall vulnerability (CVE-2026-15409), an unauthenticated attacker could make the SMA1000 appliance query internal IP addresses, cloud metadata endpoints, or backend services that are otherwise firewalled from the public internet. What makes SSRF especially dangerous is that the requests originate from a trusted internal source — the victim's own infrastructure — so they bypass network controls that would normally block an outsider.
📖 Glossary: UEFI · Secure Boot · SSRF · ADFS · Bootkit · OFAC · Cryptor · Patch Tuesday ▶
UEFI — Unified Extensible Firmware Interface; the modern replacement for BIOS that initialises hardware and hands control to the operating system bootloader at startup.
Secure Boot — A UEFI security standard that cryptographically verifies each boot component is signed by a trusted authority before allowing it to execute, preventing unauthorised code from loading before the OS.
SSRF — Server-Side Request Forgery; an attack class where an attacker tricks a server into making outbound HTTP requests to internal or external resources on the attacker's behalf.
ADFS — Active Directory Federation Services; Microsoft's single sign-on service that issues authentication tokens allowing users to log in to multiple applications with one set of corporate credentials.
Bootkit — A type of rootkit that infects the Master Boot Record or UEFI firmware, making it persistent across OS reinstalls and nearly invisible to standard security tools that only scan after the OS has loaded.
OFAC — Office of Foreign Assets Control; the US Treasury bureau that administers and enforces economic and trade sanctions against targeted foreign countries, individuals, and entities.
Cryptor — A tool purchased by malware authors to obfuscate or encrypt their malicious code so that antivirus and endpoint security products fail to recognise it as harmful.
Patch Tuesday — Microsoft's monthly security update release cycle, occurring on the second Tuesday of each month, when the company publishes fixes for vulnerabilities across Windows, Office, and related products.
