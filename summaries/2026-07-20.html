CISA Adds Exploited SharePoint RCE Zero-Day CVE-2026-58644 to KEV — Federal Deadline Was Yesterday
🔍 What happened: Microsoft SharePoint Server contains a critical deserialization vulnerability (CVSS 9.8) that attackers weaponized as a zero-day before Patch Tuesday patches shipped on July 14, prompting CISA to add it to the Known Exploited Vulnerabilities catalog with a federal remediation deadline of July 19.
👤 Who's affected: Any organization running on-premises SharePoint Server 2016, 2019, or Subscription Edition — including government agencies, law firms, healthcare, and enterprises that haven't applied the July 14 Patch Tuesday update.
✅ What to do: Apply the July 2026 Patch Tuesday update for SharePoint Server immediately; simultaneously patch CVE-2026-56164 as attackers are chaining both flaws to steal IIS machine keys and achieve persistent code execution.
🎯 ATT&CK: [Initial Access] T1190 · Exploit Public-Facing Application
Attackers exploited this unauthenticated deserialization flaw as a zero-day to gain initial footholds on enterprise SharePoint deployments before Microsoft released patches on July 14, establishing persistence via IIS machine key theft.
▶ Full analysis & beginner explanation

CVE-2026-58644 is a deserialization of untrusted data flaw in SharePoint Server's request-processing pipeline. When SharePoint deserializes crafted HTTP request data, it can be tricked into executing attacker-controlled code without any authentication. Unlike typical privilege escalation bugs, this vulnerability grants immediate code execution as the SharePoint application pool account — a powerful position from which attackers escalate further.

Post-exploitation forensics revealed that threat actors are extracting IIS machine keys from compromised servers. These keys allow forging of encrypted ViewState tokens, enabling persistent access that survives patching — a technique previously associated with sophisticated state-linked actors targeting government SharePoint deployments. CISA's advisory explicitly warns of ongoing post-exploitation persistence before the patches ship.

The federal July 19 deadline has now passed. Private-sector organizations should treat this as a P0 emergency: unpatched SharePoint instances connected to the internet are actively being compromised. Organizations should also check IIS logs for anomalous machine key access dating back to early July when exploitation likely began.

🎓 For beginners: SharePoint is Microsoft's intranet and document-sharing platform used by millions of companies. "Deserialization" is when software converts data from a compressed format back into usable objects — like unpacking a box. This bug means SharePoint unpacks a maliciously crafted "box" sent by a hacker, and in doing so accidentally runs the attacker's instructions. Critically, the hacker doesn't even need a password to send this box, making this an extremely dangerous flaw.
CVE-2026-58644 · CVSS: 9.8/10 — Maximum exploitability: unauthenticated remote code execution over the network with no user interaction required.

LegacyHive: Unpatched Windows Zero-Day PoC Grants Admin Access on All Fully Patched Systems
🔍 What happened: Researcher Nightmare Eclipse released LegacyHive, a working proof-of-concept exploit for an unpatched privilege escalation vulnerability in the Windows User Profile Service that elevates any standard user to administrator on all supported versions of Windows 10, 11, and Server — including fully patched July 2026 systems.
👤 Who's affected: Every organization running Windows 10, Windows 11, Windows Server 2016, 2019, or 2022 — no mitigation patch currently exists, and the publicly available PoC lowers the bar for exploitation significantly.
✅ What to do: Monitor Windows Event logs for anomalous User Profile Service activity (Event ID 1511, 1515, 1530); restrict local user logon access on sensitive systems; apply Microsoft advisory mitigations as they become available; watch for patch out-of-band.
🎯 ATT&CK: [Privilege Escalation] T1068 · Exploitation for Privilege Escalation
LegacyHive abuses Windows User Profile Service's arbitrary registry hive loading to allow a standard local user to mount an administrator's hive under their own registry root, gaining read/write access to privileged settings on fully patched systems.
▶ Full analysis & beginner explanation

LegacyHive targets a fundamental design decision in how Windows manages user profile registry hives. When a user logs on, the User Profile Service (UProSvc) loads their registry hive — a database of settings and configuration — and mounts it under HKEY_CURRENT_USER. The vulnerability lies in insufficient validation of which hives the service is allowed to load, permitting a low-privileged user to redirect this mechanism toward an administrator's hive file.

The researcher, Nightmare Eclipse (also identified in some reports as Chaotic Eclipse), has a documented pattern of releasing Windows exploits after disputes with Microsoft over disclosure timelines. The publicly released PoC requires knowledge of another user's credentials as a workaround, but the researcher states an unrestricted version exists. Even the constrained PoC is dangerous in corporate environments where service account credentials are often accessible.

No CVE has been assigned, and no out-of-band patch is yet available. Microsoft has not publicly commented on timeline. Security teams should treat this as a local privilege escalation risk that could be chained with remote exploits — for example, an attacker who gains initial access via phishing can immediately escalate to admin using LegacyHive while waiting for a patch.

🎓 For beginners: Every Windows user has a personal "registry hive" — a hidden database that stores their settings, like a personal filing cabinet in Windows. Normally, your filing cabinet is only accessible by you and administrators. LegacyHive tricks Windows into putting an administrator's cabinet inside yours, where you can now read and write it with your normal user powers. It's like convincing a building's mailroom to deliver a manager's locked safe to your desk by mistake.
No CVE assigned · CVSS: pending — Unpatched local privilege escalation affecting Windows 10, 11, Server 2016/2019/2022 fully updated as of July 14, 2026.

Microsoft July Patch Tuesday: Two Actively Exploited Zero-Days in SharePoint and AD FS Among Record 622 CVEs
🔍 What happened: Microsoft's July 2026 Patch Tuesday patched a record 622 vulnerabilities including two actively exploited zero-days: CVE-2026-56164 (SharePoint unauthenticated network EoP, allowing machine key theft) and CVE-2026-56155 (AD FS token-forging privilege escalation, discovered by Microsoft DART during an active incident response).
👤 Who's affected: Organizations running on-premises SharePoint Server and Active Directory Federation Services — the combination of both flaws gives attackers persistent, authenticated access across an enterprise's entire federated identity infrastructure.
✅ What to do: Apply Patch Tuesday updates immediately for both products; audit AD FS token logs for anomalous token issuance; review SharePoint IIS logs for machine key access; treat both as P0 patches alongside CVE-2026-58644.
🎯 ATT&CK: [Privilege Escalation] T1134 · Access Token Manipulation
Attackers exploited the AD FS flaw to forge authentication tokens while chaining CVE-2026-56164 to steal IIS machine keys from SharePoint, enabling persistent elevated access that survives password resets across federated enterprise environments.
▶ Full analysis & beginner explanation

The pairing of CVE-2026-56164 and CVE-2026-56155 represents a particularly dangerous exploitation chain in enterprise Microsoft environments. CVE-2026-56164 allows an unauthenticated attacker to escalate privileges on SharePoint Server over the network — no credentials required — and in documented wild exploitation, attackers used this to extract IIS machine keys. These keys enable the attacker to forge encrypted ASP.NET ViewState objects, creating a persistence mechanism that continues to work even after patching.

CVE-2026-56155 was discovered by Microsoft's Detection and Response Team (DART) during an active intrusion, confirming real-world exploitation before the patch was available. The AD FS flaw allows forging of SAML authentication tokens — the credentials used across federated Single Sign-On systems. An attacker with forged tokens can authenticate to any service integrated with the organization's AD FS server without ever needing a valid user password.

The July Patch Tuesday itself is historically significant: 622 CVEs in a single month exceeds any previous record, a volume Microsoft attributes partly to a new AI-powered internal vulnerability scanner. The sheer volume creates triage challenges for defenders, making it more likely that critical fixes get delayed behind lower-priority patches.

🎓 For beginners: AD FS is like a master key card system for company software. Instead of logging into each app separately, employees use one login that works everywhere. CVE-2026-56155 lets attackers create fake master key cards that look completely legitimate, so they can walk into any company app. Meanwhile, the SharePoint bug (CVE-2026-56164) lets them steal the "signing key" used to validate those cards — so even after Microsoft patches it, the stolen key can keep forging valid access indefinitely.

Threat actor "888" obtained source code repositories from Accenture's Azure DevOps environment containing Azure PATs, SSH keys, and RSA keys embedded in configuration files and code — a classic case of unsecured credentials in source repositories enabling lateral movement.

Azure DevOps repositories often serve as central hubs where developers store not just code but deployment scripts containing credentials, API keys, and access tokens that were "temporarily" hardcoded — and never cleaned up. Critical NGINX Vulnerability CVE-2026-42533 Can Crash Workers and Enable Remote Code Execution
🔍 What happened: F5 patched CVE-2026-42533, a critical heap buffer overflow in nginx's map module triggered by specially crafted HTTP requests; it reliably causes denial of service, and researchers argue it can enable remote code execution by supplying its own ASLR bypass — more dangerous than F5's initial advisory suggests.
👤 Who's affected: All deployments of nginx stable prior to 1.30.4 and mainline prior to 1.31.3, including NGINX Plus before 37.0.3.1 — nginx powers approximately 34% of the world's web servers and is embedded in countless load balancers and CDN products.
✅ What to do: Update nginx to 1.30.4 (stable) or 1.31.3 (mainline) immediately; NGINX Plus users should upgrade to 37.0.3.1; verify your nginx config uses the map module with regex captures before assuming non-exposure, as many default configs do.
🎯 ATT&CK: [Execution] T1203 · Exploitation for Client Execution
Specially crafted HTTP requests exploit a heap buffer overflow in nginx's map-module two-pass string evaluation, enabling unauthenticated attackers to crash worker processes (DoS) or achieve code execution on servers where ASLR protections are absent or bypassable.
▶ Full analysis & beginner explanation

CVE-2026-42533 lives in nginx's scripting engine — the code that constructs response strings from configuration directives at request time. It only triggers under a specific but common configuration: a regex-based map directive whose output variable is then referenced in a string expression following a regex capture group. The vulnerability is a classic two-pass measurement bug: the first pass measures the required buffer size, allocates memory, and the second pass writes the data — except that a regex capture value can alter the needed size between passes, causing the second pass to write beyond the allocated heap buffer.

F5's advisory conditions remote code execution on ASLR being disabled or bypassable. However, independent researchers demonstrated on a default Ubuntu 24.04 build that the overflow itself — due to its controlled nature and reverse-order memory clobbering — can recover the memory addresses needed for a payload with a single unauthenticated GET request, effectively providing its own ASLR bypass. This makes the RCE scenario significantly more realistic than F5's initial language implied.

As of July 20, no public exploit code has appeared and the vulnerability is not on CISA's KEV. However, the presence of detailed technical write-ups means weaponization is likely within days to weeks. Organizations should treat this as a patch-now situation rather than a "monitor and patch soon" one — particularly for internet-facing nginx deployments handling untrusted HTTP input.

🎓 For beginners: nginx is the web server software that handles incoming visitors for roughly a third of all websites. Think of it as a traffic controller that reads instructions about how to respond to each visitor. CVE-2026-42533 means that if you send nginx a carefully crafted request, you can overflow a memory buffer — like overfilling a box until it spills into the next one — potentially letting you write your own instructions into the traffic controller's brain. The attacker needs no login; they just need to be able to reach the web server with an HTTP request.
CVE-2026-42533 · CVSS: 9.0/10 — Critical heap buffer overflow; DoS confirmed, RCE possible under conditions that researchers argue are more accessible than vendor guidance states.

💡 Today I Learned: Registry Hive Hijacking ▶
Every Windows user has a personal "registry hive" — a hidden binary database file that stores all of their settings, preferences, and application configurations, loaded into memory when they log in. Registry hive hijacking exploits flaws in how Windows loads these files, tricking the operating system into mounting one user's hive under another user's registry namespace — effectively letting a low-privileged user read and write the settings database of an administrator. This technique is particularly tricky to patch because it often abuses legitimate Windows design behaviors (like User Profile Service hive loading) rather than traditional memory corruption bugs, making the fix a delicate balance between security and breaking normal account-switching functionality.
📖 Glossary: ASLR · IIS · EoP · CVSS · PAT · HIPAA · RCE ▶
ASLR — Address Space Layout Randomization: a memory security technique that places program code and data at random locations in RAM each time a process runs, making it harder for attackers to predict where to direct a memory-corruption exploit.
IIS — Internet Information Services: Microsoft's web server software bundled with Windows Server that hosts web applications like SharePoint; IIS machine keys are cryptographic secrets it uses to sign and encrypt web tokens.
EoP — Elevation of Privilege: a class of vulnerability where an attacker with limited access (e.g., standard user) exploits a flaw to gain higher permissions (e.g., administrator or SYSTEM) on the same machine or network.
CVSS — Common Vulnerability Scoring System: the industry-standard 0–10 scale used to rate vulnerability severity, where 9.0–10.0 is Critical, 7.0–8.9 is High, 4.0–6.9 is Medium, and below 4.0 is Low.
PAT — Personal Access Token: a long-lived token used instead of a password to authenticate automated scripts and tools to services like Azure DevOps or GitHub; if leaked, it grants the same access as the user who created it.
HIPAA — Health Insurance Portability and Accountability Act: U.S. federal law requiring organizations that handle protected health information to safeguard patient privacy and notify affected individuals within 60 days of discovering a data breach.
RCE — Remote Code Execution: a vulnerability class that allows an attacker to run arbitrary commands on a target system over a network connection, typically without physical access or prior authentication.
