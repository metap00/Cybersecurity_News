mcp-pinot: Unauthenticated Remote Tool Invocation via Default oauth_enabled=False + 0.0.0.0 Bind (CVE-2026-49257)
🔍 What happened: The mcp-pinot MCP server shipped with OAuth disabled by default and bound to all network interfaces (0.0.0.0), allowing anyone on the network — no credentials required — to invoke any Apache Pinot database tool, including queries and schema modifications.
👤 Who's affected: Organizations running mcp-pinot (the Python MCP server for Apache Pinot) before version 3.1.0, particularly any deployment exposed beyond localhost such as in Kubernetes or Docker environments.
✅ What to do: Upgrade to mcp-pinot v3.1.0 immediately, which defaults to 127.0.0.1 binding and requires OAuth before any non-loopback exposure; if you cannot upgrade, firewall the server and restrict access to localhost only.
▶ Full analysis & beginner explanation

mcp-pinot is a Model Context Protocol server that bridges AI assistants (such as Claude or GPT-based tools) to Apache Pinot, a real-time distributed OLAP datastore used in analytics pipelines. The vulnerability stems from two insecure defaults: oauth_enabled=False by default, meaning no authentication was enforced, and the HTTP server bound to 0.0.0.0, meaning it accepted connections from all network interfaces — not just localhost.

This combination produced a perfect CVSS 10.0: network-accessible, no authentication required, no user interaction needed, with full scope impact on confidentiality, integrity, and availability. An attacker on the same network (or any network if the server was internet-facing) could invoke any registered MCP tool — running SQL queries, modifying Pinot schemas, or fetching sensitive analytics data — with zero friction.

The fix was merged in PR #95 and released as v3.1.0 on May 25. The advisory was published June 26, 2026. The patch changes the default bind host to 127.0.0.1, requires OAuth to be explicitly enabled before any non-loopback exposure, makes Helm chart exposure opt-in and OAuth-gated, and adds input validation to the argument parser.

🎓 For beginners: MCP (Model Context Protocol) is how AI assistants connect to tools and databases. Imagine leaving the door to your database server wide open with a sign saying "no password needed." That is exactly what mcp-pinot did by default — it listened for commands from anyone on the network without checking who they were. Anyone nearby could have read, modified, or deleted your analytics data just by sending the right request.

Nezha Dashboard: Cross-Tenant Terminal and File Manager Session Hijack via Unvalidated WebSocket UUID
🔍 What happened: In Nezha v1.14.13–v1.14.14 and v2.0.0–v2.0.9, any authenticated dashboard user can hijack another user's active terminal or file-manager WebSocket session simply by knowing the session's stream UUID, as the server checks authentication but never verifies session ownership.
👤 Who's affected: Teams and hosting providers using Nezha as a multi-user server monitoring and management dashboard — particularly those where different users manage different servers from the same dashboard instance.
✅ What to do: No patch is available yet; restrict Nezha dashboard access to fully trusted administrators only, and avoid multi-tenant deployments until a fix is released.
▶ Full analysis & beginner explanation

Nezha is a popular open-source server monitoring platform that gives administrators a web dashboard to view metrics, open terminals, and manage files on remote servers. The terminal and file-manager features work over WebSocket — persistent real-time connections identified by a UUID (a unique session ID) generated when the session is created.

The vulnerability is a classic authorization failure: the WebSocket endpoints GET /ws/terminal/:id and GET /ws/file/:id verify that the requester is a logged-in user, but they never verify that the logged-in user actually owns the session referenced by :id. Any dashboard user — even a low-privilege "RoleMember" — who obtains a live stream UUID can attach to that session and gain interactive shell access or full file-manager control on the target server. This is cross-tenant access: a user on Server A can compromise Server B's terminal if they learn the UUID.

Scoring CVSS 9.9 (network, low complexity, low privilege, no user interaction, changed scope, high impact across all three CIA dimensions), this flaw is rated critical. The UUID can potentially be predicted, brute-forced (if short), observed in logs, or obtained via social engineering. No patch exists at time of publication.

🎓 For beginners: Think of a hotel where every room has a unique door code. This bug is like the hotel giving you a master key as long as you know any room's code — even if that room belongs to a different guest. In Nezha, each server management session gets a unique ID (like a room code), but the system never checks whether you own the session you're trying to open. A malicious user could take over any other user's server terminal.
No CVE assigned yet · CVSS: 9.9/10 — Authenticated low-privilege user can hijack any active server terminal across tenant boundaries; no patch available.

deepstream Server: Prototype Pollution Enables Privilege Escalation from Any Authenticated Write User (CVE-2026-49252)
🔍 What happened: deepstream server versions up to 10.0.4 are vulnerable to prototype pollution — an attacker with write access to any record can inject malicious properties into JavaScript's base Object prototype, potentially escalating privileges to administrator level.
👤 Who's affected: Applications using @deepstream/server for real-time data synchronization, gaming backends, collaborative apps, or IoT platforms where users with write permissions may be untrusted or partially trusted.
✅ What to do: Upgrade to deepstream v10.0.5 immediately; as a temporary workaround, filter all incoming messages containing the paths __proto__, constructor, or prototype before they reach the server's message pipeline.
▶ Full analysis & beginner explanation

deepstream is a real-time data server used to synchronize state across clients — common in collaborative editing, multiplayer games, dashboards, and IoT applications. Its core concept is a "record" — a shared, mutable JSON object that any permitted client can read or write to.

The prototype pollution vulnerability means that when a client writes to a record using a specially crafted path like __proto__.isAdmin or constructor.prototype.role, deepstream processes the path without sanitization and mutates JavaScript's base Object.prototype. Because every JavaScript object inherits from Object.prototype, this mutation affects all objects on the server — including internal authorization objects. The result can be privilege escalation: the attacker's session may gain permissions they were never granted, up to and including administrative control.

CVSS 9.9 reflects the near-universal impact: once prototype pollution succeeds, server-wide state is compromised. The fix (v10.0.5) adds path sanitization before record mutations are applied. The recommended workaround — blocking messages with __proto__, constructor, and prototype in paths at the network layer — is effective but must be applied before message parsing to prevent exploitation.

🎓 For beginners: In JavaScript, every object shares a common ancestor called "Object.prototype" — like a template all objects copy from. Prototype pollution is when an attacker tricks the server into adding properties to that shared template, which then magically appears on every object in the program. It is like sneaking a skeleton key onto a key ring that all the locks in a building are based on — suddenly every door is open.
CVE-2026-49252 · CVSS: 9.9/10 — Authenticated user with record-write permission can pollute the global Object prototype, enabling server-wide privilege escalation. Fixed in v10.0.5.

Incus Container Runtime: Malicious Image Server Triggers Arbitrary File Write and Root Code Execution (CVE-2026-48769)
🔍 What happened: The Incus container daemon trusts the Incus-Image-Hash HTTP response header from image servers without validation, allowing a malicious image server to control the file path used during image download and write arbitrary content to arbitrary locations on the host filesystem, leading to RCE as root.
👤 Who's affected: Operators using Incus (the LXC/LXD successor for Linux containers and VMs) to pull images from external or third-party image servers, especially automated pipelines that fetch images without manual oversight.
✅ What to do: No patch is available yet; restrict source.type=url image pulls to a curated allowlist of fully trusted servers and avoid pulling images from any untrusted source until fixed.
▶ Full analysis & beginner explanation

Incus is an open-source system for managing Linux containers and virtual machines, used in cloud hosting, development environments, and infrastructure automation. When a user creates a container from a URL-based image source (source.type=url), the daemon issues an HTTP HEAD request to the image server to discover the image hash and download URL via response headers.

The vulnerability lies in cmd/incusd/images.go: the daemon reads the Incus-Image-Hash header value from the server's response and uses it directly as the Alias parameter — without any sanitization or path validation — when constructing the local file path for the downloaded image. A malicious image server can return a crafted hash value containing path traversal sequences (e.g., ../../etc/cron.d/backdoor), causing the daemon to write attacker-controlled content to any location on the host filesystem. Since incusd typically runs as root, this constitutes arbitrary root file write, which trivially leads to root code execution (e.g., via cron, authorized_keys, or LD_PRELOAD).

With a CVSS score of 9.9 and a Changed Scope, the blast radius extends beyond the container daemon itself to the entire host system. No patch has been released at time of publication.

🎓 For beginners: When you download a container image, the software asks the image server "where should I save this file?" This bug means a rogue image server can answer "save it here: /etc/cron.d/evil" — and the software obeys. Since the container manager runs with full administrator (root) privileges, the attacker can then make the computer run any command they want just by setting up a fake image server.
CVE-2026-48769 · CVSS: 9.9/10 — Malicious image server achieves root arbitrary file write on the host via an unsanitized HTTP header; no patch available.

Relyra SAML Library: Signatures Never Cryptographically Verified — Authentication Bypass in All Versions (CVE-2026-49454)
🔍 What happened: The Erlang SAML library Relyra (versions 1.0.0 and 1.1.0) accepted forged SAML authentication assertions because it never actually performed cryptographic signature verification — the SignatureValue field was present and parsed but never validated against the configured identity provider's public key.
👤 Who's affected: Applications using Relyra for SAML-based Single Sign-On (SSO) authentication, particularly Erlang/Elixir services integrated with enterprise identity providers like Okta, Azure AD, or PingFederate.
✅ What to do: No patch is available; immediately disable Relyra-based SAML authentication and fall back to an alternative auth method — any account protected only by Relyra-mediated SAML should be treated as potentially compromised.
▶ Full analysis & beginner explanation

SAML (Security Assertion Markup Language) is a protocol used in enterprise SSO: an identity provider (like Okta or Azure AD) issues a digitally signed XML assertion saying "this user is authenticated." A service provider (your app) must verify that signature before trusting the assertion — that verification is the entire security boundary of SAML.

Relyra's critical flaw: in versions 1.0.0 and 1.1.0, the library parsed the SAML XML structure, extracted the SignatureValue element, but never called :public_key.verify against the canonicalized SignedInfo using the configured IdP certificate's public key. Similarly, DigestValue was never recomputed over the canonicalized referenced element. The result: any attacker can construct a completely fabricated SAML assertion (with arbitrary user identity, roles, and attributes), include a fake or empty SignatureValue, and Relyra will accept it as valid authentication.

This is a total authentication bypass — CVSS 9.1. An attacker who knows the SAML endpoint URL can log in as any user, including administrators, without possessing any credentials. No patch is available at time of disclosure.

🎓 For beginners: SAML is like a signed letter from HR saying "this person is an employee." The signature proves HR wrote it and nobody tampered with it. Relyra's bug is that it read the letter and trusted it without ever checking whether the signature was real — like a bouncer who glances at the wristband but never actually scans it. Anyone who forged a letter, even with gibberish for a signature, would be let right in.
CVE-2026-49454 · CVSS: 9.1/10 — Total SAML authentication bypass; forged assertions are accepted as valid. Affects all Relyra versions. No patch available.

Nezha Monitoring: Pre-Auth Path Traversal via /dashboard.. Prefix Confusion Leaks Master JWT Secret (CVE-2026-53519)
🔍 What happened: Nezha's dashboard frontend handler uses a string prefix check (strings.HasPrefix(url, "/dashboard")) to decide whether to serve admin assets, but the check matches /dashboard.. — allowing an unauthenticated attacker to request /dashboard../data/config.yaml, which resolves to the server's config.yaml containing the jwt_secret_key.
👤 Who's affected: Anyone running an internet-exposed Nezha monitoring dashboard — once an attacker has the jwt_secret_key, they can forge admin-level JWTs and take over the entire dashboard without credentials.
✅ What to do: No patch is available yet; immediately firewall your Nezha dashboard from the internet, restrict to VPN or internal network access only, and rotate the jwt_secret_key if the dashboard was ever internet-accessible.
▶ Full analysis & beginner explanation

This is a path traversal vulnerability rooted in a subtle string-matching mistake. Nezha's NoRoute fallback handler is supposed to serve the admin frontend for any URL that starts with /dashboard (meaning URLs like /dashboard/overview or /dashboard/settings). The check uses strings.HasPrefix(url, "/dashboard") — which also matches /dashboard.. because the string /dashboard.. does indeed start with /dashboard.

When the handler strips the /dashboard prefix using strings.TrimPrefix, the remainder is ../data/config.yaml. This gets joined with the admin frontend asset directory via path.Join("admin-dist", "../data/config.yaml"), which Go's path normalizer resolves to data/config.yaml — the server's configuration file. This file is then served verbatim by http.ServeFile without any authentication check, because the handler never reaches the authentication middleware (it was treated as a frontend asset request).

The config.yaml file typically contains jwt_secret_key — the HMAC secret used to sign all JWTs for dashboard authentication. With this key in hand, an attacker can forge a JWT claiming any identity (including administrator) and authenticate to the dashboard without credentials. Combined with the GHSA-q6xx WebSocket session hijack also published today, a Nezha dashboard compromise is now trivial end-to-end.

🎓 For beginners: A JWT secret key is like the master stamp used to certify every login token. If someone steals that stamp, they can create fake tokens that say "I am the administrator" — and the system will believe them. This bug lets anyone, without logging in, trick the server into handing over that master stamp just by using a slightly misspelled web address (/dashboard.. instead of /dashboard).
CVE-2026-53519 · CVSS: 9.1/10 — Pre-authentication path traversal leaks jwt_secret_key; enables complete admin takeover. No patch available.

💡 Today I Learned: Prototype Pollution ▶
In JavaScript, every object secretly inherits from a shared ancestor called Object.prototype — think of it as a master blueprint that all objects copy their default behaviors from. Prototype pollution is an attack where a malicious input tricks an application into adding or overwriting properties on that master blueprint, causing the injected properties to silently appear on every object in the entire running program. 📖 Glossary: CVE · CVSS · MCP · SAML · JWT · IDOR · Path Traversal · Prototype Pollution · RCE · Supply Chain Attack · WebSocket · OAuth · SSRF · XSS · DoS ▶
CVE — Common Vulnerabilities and Exposures; a standardized identifier (e.g. CVE-2026-49252) assigned to each publicly known security vulnerability.
CVSS — Common Vulnerability Scoring System; a 0–10 numeric scale rating vulnerability severity, where 9.0–10.0 is Critical, 7.0–8.9 is High, 4.0–6.9 is Medium, and 0.1–3.9 is Low.
MCP — Model Context Protocol; an open standard that lets AI assistants securely connect to external data sources, tools, and databases in a structured way.
SAML — Security Assertion Markup Language; an XML-based protocol that lets identity providers (like Okta or Azure AD) pass signed authentication assertions to service providers so users only log in once (SSO).
JWT — JSON Web Token; a compact, digitally signed token used to securely transmit identity claims between parties — signing requires a secret key that, if stolen, lets anyone forge valid tokens.
IDOR — Insecure Direct Object Reference; a vulnerability where user-supplied input (like an ID or filename) is used to access objects directly without verifying the requester owns or is authorized to access that object.
Path Traversal — An attack that uses sequences like ../ in file paths or URLs to navigate outside the intended directory and read or write sensitive files on the server.
Prototype Pollution — A JavaScript attack where malicious input injects properties into Object.prototype, the shared ancestor of all JS objects, causing those properties to silently appear on every object in the application.
RCE — Remote Code Execution; a class of vulnerability allowing an attacker to run arbitrary commands on a victim's system over a network, typically the most severe outcome of exploitation.
Supply Chain Attack — A cyberattack targeting a trusted software dependency or build tool rather than the victim directly — the attacker compromises something the victim installs or trusts.
WebSocket — A communication protocol providing full-duplex (two-way simultaneous) communication over a single TCP connection, commonly used for real-time features like live terminals and chat.
OAuth — An open standard for access delegation that lets users grant applications limited access to their accounts on other services without sharing passwords.
SSRF — Server-Side Request Forgery; an attack that forces a server to make HTTP requests to attacker-specified targets, often used to probe internal services or cloud metadata APIs.
XSS — Cross-Site Scripting; an attack where malicious JavaScript is injected into a trusted web page and executed in other users' browsers, enabling session theft, credential harvesting, or page defacement.
DoS — Denial of Service; an attack that makes a system or service unavailable to legitimate users, typically by exhausting CPU, memory, connections, or other resources.
